我的设计博客

为了我的设计博客，前几天建的Lofter博客，今天终于布置好了。

不得不说，网易的Lofter轻博客做得很好，各种自定义功能与模块,管理也很方便快捷。其中博客模板就有很多，而且很漂亮，不过不能随意修改，只能通过其设定好的自定义项修改。

今天挑来挑去，不是差这个不够理想，就是差那个不够美观。最后，终于确定了一个，响应式设计的，没有什么重大缺点也没有什么亮点的模板。

博客名字叫伴星流形，自己还是有点文学情节，起了这个名字，伴星是我一直以来用的网名，流形是源于文天祥的《正气歌》：

天地有正气，杂然赋流形。下则为河岳，上则为日星。

另外还有易经《彖》：

大哉乾元，万物资始，乃统天。云行雨施，品物流形。

流形，字面上解释起来就是：各种形态，指天地宇宙间的一切形体。

一经看到这个词，我就喜欢上了它了，流形流形，设计不就是这样吗？上善若水，大道无边。任何领域从技术升到艺术层面就应该不再有约束存在了，这和我高一的语文老师说的差不多，原话忘了，大致意思是：有些同学，写的作业以为自己在写狂草，字难看得我看都不想看。你们现在写字就要一个一个写工整，等你达到更高境界了，才能随心所欲，任笔挥写，讲求意到笔随。

所以博客名叫伴星流形，英文名取流形翻译叫Manifold。愿我的设计能有星空的虚空磅礴，亦有流形般不受约束，尽情绽放自己的创想。

但愿如此。

2013年小小总结

又一年了，小小总结下。

在12年1月正式学习flash开发，13年1月还是那么满怀期待的尝试flash新技术（starling、增强现实），14年1月了，我开始认真关注设计圈子。

是的，我“转行”了。

严格上说不叫转行，只是我将重心天平大大的移到设计上。一直以来对设计的兴趣没有让我落下。零零碎碎的，只是缺少能拿的出手的作品。

2013年的历程：

  • 3月开始成为工作室负责人之一。与庆杰一起带新一届的维生数。
  • 4月策划带人做手游Pulupulu（目前停留于那时完成的测试版）
  • 5月参加学校社团组织的平面动画设计大赛（动画一等）
  • 7月暑期实践学习android开发新生网app（失败）
  • 10月开发桌面应用GIMS（目前版本v0.6.5）
  • 11月带队参加“规划杯”比赛（优秀奖）

特别的感受呢，有三方面：做工作室负责人，带项目，参加比赛。

做工作室负责人。我和庆杰带的这届，到现在做得也算是足够了。刚接手时，那几乎是工作室的最低潮点，服务器垮了之后焦头烂额没弄好，交给了我们。那时，服务器坏了，工作室主页不在了，博客没了，logo丢了原图，工作室缺少了载体，人也散了。从今年3月接手以来，重新建立博客、认真举办每次分享会，购置投影仪看电影，开总结会议、谈未来计划，解决服务器问题，开各种交流会，做了几个自主项目，招新也做得有范了。在不停地处理工作室各种事务、技术完善与追求中，逐渐形成了一套“独特”的管理、学习、交流方法。

但，问题还是有的，几个项目“烂尾”，以及工作室这近乎华丽的外表上，似乎预示着工作室正在崛起的路上，实则不然，细心发现，就会觉得有失初衷——大家玩在一起。上学期期末记得潇滨提醒过我们：技术是很好了，但活动不能少。

执行力与凝聚力是一直存在的问题，执行力很大方面是项目组里面需要有一个好leader。而工作室很少组织活动，因为一旦组织，难有集体响应，只能几个小团体间的活动。在工作室煮煮糖水、煮煮咖喱饭啊，真的很好吃~但才那么几次。工作室嘛，大家就要玩得起来才叫工作室~

这一年来我们追求技术太过了，太过于技术这块，而让我们在这路上丢失了很多其他东西。这之后需要我们再度拾起。

带项目
发现我的大局观越来越好了，可能是我兴趣广泛让我的眼界不断开拓，以及学会了一套适合自己的学习方法，似乎一切技术上的问题都难不倒我，“似乎”。

这一年带了三个算是项目的项目，一是手游pulupulu，4人，一个月。这算是一次带新人吧，我做开发，但我会查阅学习策划、设计、开发实现，我尝试了之后再简单地教他们策划、设计、开发。这让我发现了一旦你对这个项目的整体及各方面都掌握到你一个人只要有时间都能独立完成的程度，你是一个合格的项目负责人了，因为你对这个项目了如指掌，能够很好地确立计划与ToDo List，一步一步推进项目进度。

二是新生网App，4人，数天，我做设计和协同开发，在暑假实践要求几天完成，显然不可能，大家都懒惰得后面回家都没去完成了。烂尾！

三是GIMS，这是一次自己的独立开发，将近一个月时间。设计师给了设计稿后交流了几次后开始工作。

总结一下，项目需要一个好Leader，去爱这个项目、产品，也带着大家一起去爱。这样项目才会充满生机与活力。同时负责人需要有良好的大局观，做好规划，分配好工作，并随时应对各种变化。这样才能较好地完成一个项目。

参加比赛
一是社团那个平面设计动画大赛的动画组，几天完成了一个小动画也能拿奖，只能说明我是真的切中了要点，迎合了评委的心意。一是设计上算是创新了，二是展示也较好，三是主题鲜明积极向上。

二是规划杯这个比赛，5人，花了两个星期。这是一次策划比赛，技术含量不高，但我用带项目的管理经验去带队参加这个比赛，效果不错。规划、线上讨论、分工、进度跟进，一步一步完成下来，本以为三等前无误，结果只拿了个优秀奖，不过奖金也有4K。

参加比赛这件事，我算是较有心得的，凡是获奖机会不大的比赛我是不会参加的，(难道参加比赛只为了获奖吗？确实，细心想想你就会发现参加比赛你获得的进步并不会太大，主要是图个参与，顺带获个奖不更好吗？）而参加比赛我一定会死死抓住获奖关键点，我会去想主办方需要得到的是什么作品？评委需要看到的是什么作品？其它关系不大的事情不会太花时间与精力。哲学上这叫“主要矛盾”，策划比赛你去抠设计，设计比赛你去抓狂代码，这不是笑话吗？

过去一年平平淡淡，没有跌宕起伏经历，却也有希望与失望，有快乐与伤心，有成功与失败。

确实让我蜕变了很多，至少锐气磨减了，眼界更开阔了，看的书多了，技术也进步了许多。

嗯，一切安好，还需努力，2014年继续加油！晚安。

Morn UI使用感受

在开发AIR桌面应用GIMS的时候，想起之前在微博上的看到的MornUI，于是便开始了接触了这个有爱的MornUI

Morn UI简介
Morn UI,是一个flash轻量级，高性能，可视化UI解决方案。它包括了一个Morn
UI库和一个搭配的可视化编辑器Morn
Builder，下面是官方介绍的优势,基本上都认同：

  1. 轻量级，Morn
    UI库以精简，直观为设计理念，代码轻量，能快速上手，减少学习成本，Morn
    UI全部库总大小不到30K
  2. 可视化，Morn
    UI提供强大的可视化编辑器，布局及属性均可在编辑器直观设置，实现UI和逻辑分离，让美术和程序轻松合作
  3. 高性能，Morn
    UI设计之初就以高性能为主要目标，以位图为基础，利用延迟渲染机制，实现了高性能
  4. 易扩展，无论UI组件还是编辑器插件，都非常易于扩展，编辑器支持即改即用，轻松实现个性化
  5. 自动资源管理，Morn编辑器自动管理UI资源，以灵活的方式打包资源（jpg+alpha）
  6. 同时支持位图和矢量图，并自动生成UI代码
  7. 支持多项目，支持多视图开发，支持代码视图，支持协同开发及svn管理
  8. 支持多语言，内置语言包管理器及提取工具，快速国际化
  9. Morn比flex轻量高效，比flash

新版本已经全面支持移动设备开发了，其作者yungzhu是个热心人（在q群上~），在官博上有demo和使用教程，还有在线API文档大概一个下午就能摸熟，上手不难，但一经上手，你就会深深地爱上了它！（这是所有第一次使用的人的共识，但因为Morn

UI还在不断完善，所以随着深入开发你就会发现其坑之多，且陷入其中，“其乐融融”！）

使用感受

首先还是得感谢Morn UI带给我的“愉悦”开发过程（真心的），在近一个月的时间的GIMS开发过程中，按照规划开发一个个功能界面，都能在Morn UI库对应找到相关的组件、相关的函数可供调用。但总有一些时候会发现并没有这个实现方法，所以这个时候你就要到官方Morn交流群（官网有）里大吼一声，刚刚还在相互吹水的各位“大神”就会停下来给你解答，解答有时还会很热心的让你截图一步步找出解决办法，等没问题了，继续吹水~很喜欢这个氛围，虽然大家看起来都老不正经，但确是个热心的交流群（欢迎更多的人加入这个大家庭~）。

Morn本身UI组件库就封装好了常用的UI组件和其需要用到的功能函数，加上其可视化编辑器，对于UI布局和功能开发真的很方便，下面简单地说几点：

  1. UI组件切图。根据各个组件的不同，你需要相应的按照规定格式切出来，并按规定命名，规定格式及命名方式可以下载官方demo上面的UI资源查看或查看官方教程。一般是UI的3个状态（main、hover、pressed）按三等分上下排布，切成png。如button、checkBox、tab。（如下图）
  2. “写”界面。直接使用Morn
    Builder，将切好的UI组件“导入”，通过这个可视化编辑器，根据需要调整相应参数，导出即可生成写好的UI代码。（参照Morn
    UI教程）
  3. 写功能逻辑代码。这个我是开始贪快直接将所有功能写在页面的as里（说好的MVC呢），后来到后面发现越来越累赘，想重构成MVC模式的发现很难了T^T。结果导致代码最后达到1000多行…我的错，所以最好在最开始就将该有的相关组件UI、界面分开来（不会的还是参照官方demo源文件）。
  4. 组件功能的不足。例如，写表单时，表单输入框的文本提示功能是没有的，需要自己写获取焦点失去焦点的提示文本代码。还有在写list时发现，选中操作不好写，List类没提供相关的监听函数，需要自己写。后来发现新版本改进了，但因为已经进行到一半，试了下更新UI库，结果就出问题了，还好git还原回来了。还有其他一些坑，迟些再补充。

大概这么多，回到期末备考大军中去了，想深入了解Morn
UI还是直接看其源代码最快。

另外还有一篇别人写的不错的使用心得，对编辑器原理和界面管理的一些分析,可以看看。

尝试基于桌面端的air应用程序开发

前言

一直就打算做一个桌面软件来玩玩，自从知道air能够实现之后。这次终于有一次机会了，开发热情立马飙升！

在10月17号晚的工作室设计交流会上，跟设计师谈到了她很久以前的一个设计一直无人开发，而AIR的种种特性让我有信心开发完成，于是就\
接了下来，到周末拿到设计稿后，三个多星期的愉(ku)快(bi)开发过程就此展开了。下面记录一下感想，技术含量不多。(又是一篇流水账= =）

起始

这个设计是一个用于学校社团数字中心的一个女生节活动的相片信息管理系统，主要用到列表，数据库操作，文件操作，这些Air都能做到。

在开始的那个周末，不停尝试，开始是直接将psd导入到flash
pro，想着这样开发会不会快捷一点，界面是很容易出来了，但是里面的控件得自己写，爽了一下这个不可操作的软件后，便否定了这个方案。

后来尝试了下flex开发，用flash
builder，但很久没用的原因，在写界面的时候，卡在了一个地方，很难写，也否定了这个方案。

接着想有什么UI库可以用，便想起了之前没怎么关注的MornUI，花了一个下午的时间研究了下MornUI的开发过程，试了一下做界面，发现很简单！超级方便！立刻爱上了MornUI了，但很快便遇到问题了，加了MornUI的官方交流群问，大家都很热情解答~问题很快就解决了。关于MornUI开发过程将在下一篇文章中介绍到。

第一周

这周开始几天基本上是查阅了AIR开发文档，切图（为适应MornUI开发的特别切图），也没再麻烦设计师了，我自己搞掂了（会ps就是那么简单自信），使用MornUI自带可视化界面编辑工具Morn
Builder做界面，项目代码逻辑是在FlashDevelop上写。很快，把一个像样的界面做出来了。（幸福来得那么突然~）

这周刚好赶上电工实训，闲的时间比较多（把实验要求的任务做完后除了给老师检查基本上是自由时间了），每天都是从宿舍——饭堂——实验楼——综合楼之间往返（走路…），乐在其中，一心只想着这个项目，开发热情空前的高！

在写完大体的界面后，因为是自定义窗口，相对于使用系统的自带窗口麻烦很多，自适应大小变化、拉伸、窗口按钮（最小化、最大化、关闭）操作都得自己写。

参考了Air窗口的外观修饰和Air开发文档-窗口部分，但还是存在问题，例如最大化、窗口适应大小比那话，不停尝试困扰了一段时间后，就此放弃…

在基本写完界面，熟悉了MornUI的开发后，在周四就将其pull上github作项目管理与版本控制。这时界面左侧的工具栏按钮有点问题（后来才知道因为图案没设置好是否接受鼠标，遮挡住按钮的获取鼠标），以及中间的列表没写，只有一个大致的界面窗口。

之后按照项目规划（自己定的），周五先完成drawer控件(切图、在Morn
Builder上制作UI），其中MornUI的不熟悉还是遇到了一点问题。

周六先建立数据库，准备储存新建的信息以及作为list列表的数据。以及数据库操作方法的建立。这里参考了Adobe
AIR中的 SQLite
数据库简介和AIR连接本地数据库的CRUD实例。

好了，为了写新建信息功能，周日在写drawer新建的表单时遇到坑了，这个表单没有提示信息的方法，得自己写获取焦点、失去焦点的方法。

第二周

周一将一个完整的表单（含提示信息功能）写好了之后，周二将数据库再修整完善了一下，周三实现新建信息存入数据库功能，周四写好了list列表，并能接收到数据库的数据。

之后就是各种完善数据库操作及列表、drawer功能（不熟悉数据库的后果T^T）。

初次尝试了匹配文件功能进行文件操作，AIr很容易实现。参考：air开发文档

中间少不了一两次写到兴奋处舍不得回宿舍，直接在工作室通宵，天寒地冻的，一件短袖~

第三周

第三周因为电工实训结束了，回到了课业苦海中，开发放松了很多，闲暇的将功能写完善，包括drawer的显示详细信息功能、编辑更改原有信息功能、匹配文件功能、搜索功能，大致上就差不多完成全部了。（想象中）

第四周

一样由于接近完成，放松了很多，结果，第四周还剩余的两三个模块就是大坑，代码开始偷懒，项目没采用MVC，临时改也花了几天没重构好。

将剩余的功能完善好，就在周末的女生节活动前各种拼命完善，寻找替代功能，在周五发布软件，icon还是用我做着玩的那个，然后晚上找bug，修bug到天亮（期间还弄了个介绍并提供在线安装的网页…）

实际应用

还好啦，在11月16日周六早上，看着真正投入使用就像自己孩子出生一样欣慰，但早上使用中反映各种bug（在我电脑上测试时明明没的！），上午再修改了下，中午带着疲惫的双眼到活动摊位玩了一下，看看实际使用中的情况，接着就回去安心地补眠了~

晚上将剩余的导出表格功能写好，第二天也比较轻松，下午在女生节活动里玩了很久~晚上写数据库合并功能又失败了，后来是采用人工合并的替代方案…

后记

我将这次视为封山之作…显然还不算，我的\
flash开发还没学到位，编程模式还是渣，基础也不行了。但没办法，紧接着下学期找实习了，flash这块现阶段是基本上找不到好实习的。在做这个air应用的过程中也发现自己的编程短板（难道真的有天赋一说！？），只能转方向了，学的比较杂，但从往至今一直热爱的设计的确是我的一大长项和算是比较有天赋的，一直以来也没停止在设计上的追求。

一想到往后的日子就要专心于设计了，很舍不得flash和编程…

感慨一下，flash在几年前有了air之后是多么强大啊！但，现在终究是敌不过退出历史舞台的命运。我想主要一个原因是在10年air刚刚火时大家都没有抓住机会，缺少包括app开发、运营、用户在内的整个生态圈的建立。还有一个原因当然就是adobe公司的策略改变，

Actionscript3事件调度与事件流

在做pulupulu游戏时，我再次研究了下，以前一直没认真搞懂的事件流（Event Flow）。发现以前真的很无知，对于事件（Event）。

首先，AS3中，事件可分为两类：内建事件，它描述运行期环境的状态变化；自定义事件，它描述一个程序的状态变化。内建事件包括用户输入、点击鼠标等事件，而自定义事件则更广泛，可能是一个游戏的开始或一次文本的提交。

为了处理（Handle)或响应事件，使用事件侦听器（Event Listener）。事件侦听器是一个函数或方法，它注册是为了等待一个事件发生时执行。当一个给定的事件已经发生时为通知一个程序，as执行所有已经注册了该事件的事件侦听器，这个通知过程称为一个事件调度（Event DIspatch）。

当as以一个对象为目标调度一个事件时，如果该对象不是一个显示层次结构的一部分，则那个目标是被通知该事件的唯一对象。相对而言，如果该对象是一个显示层次结构的一部分，则事件对象将会在这个显示列表中“穿行”，直到抵达事件目标，它不仅通知了那个目标同时也通知了一个显示列表的对象。这便是形象的事件流（Event Flow）。

如上所言，事件调度分为三个阶段：捕获阶段（Capture Phase），目标阶段（Target Phase），冒泡阶段（Bubbling Phase）。

捕获阶段，目标的同显示列表里的元素都会被通知该事件已发生。这是一个向下探的过程，在探到目标之前路过其他东西都会被摸到。

目标阶段，as通知目标对象该事件发生了。

对于一些事件*，事件调度到此就结束了。但对于其他事件，将继续前进到第三阶段——

冒泡阶段，目标的同显示列表的对象被通知该目标已成功接收到事件通知。

似乎只需要捕获及目标阶段就可以了，但既然存在三个阶段，当然是为了适应更多的开发需求。

三个阶段就如捕鱼的——撒网、捕捉、收网~不同的是这里撒网只为捕捉一条鱼。

值得注意的是框架Starling的事件没有“捕获”的阶段。这些事件只是简单的被一个对象派发，并可以选择 冒泡。他们不能向相反的方向传递。当派发事件（dispatchEvent）时，即派发事件到所有注册了这个事件类型侦听器的对象。如果一个事件开启了”冒泡”，它会沿着父级关系，一直向上，直到碰到了根节点，或者中间被手动停止它的传递。

*四种事件类型只有一个目标阶段——Event.ACTIVATE、Event.DEACCTIVATE、Event.ENTER_FRAME、Event.RENDER。

Pulupulu手机游戏开发记录（二）——技术篇

第一次制作手机游戏，总的来说，给我的感觉还是不错的，从策划到美工，从设计到程序，其中遇到的困难也不少，这些困难包括技术上的，也包括团队上的。虽然同一个工作室，但自发团队免不了拖拉问题，一拖拉就拖慢了整个项目的进度，更严重的拖了太久就减淡整个团队的开发热情。另外还因为各自的空闲时间不一样，很难聚在一起，最多的只是在讨论组里讨论。
一个项目没了热情，结果就注定是个坑。才刚做到了beta0.1，后面还有东西要加。

技术上的话，困难也不大，但还是再做个技术相关的总结吧，横跨四个领域的总结！

一、策划

这次的策划是平面组的bobo和ck，虽然说是策划，但他们也不懂得该怎么做。

“策划也不懂该干什么，相对于策划，其实我更希望能做有关设计或者画画的东西，只是能力不够而已。”——bobo

作为项目总监，我根据我已知的制作游戏项目需要，分配给他们策划的任务：将游戏流程转化为“线框图”，整体完善游戏机制、玩法。其中还包括一些小讨论：输赢机制、游戏道具的添加、有新意的玩法等等。

将游戏流程转化为线框图，这很重要，即对游戏策划的整体框架的确立，也帮助了程序员架构游戏程序。制作时是参考江南大学交互工作室的一些交互稿的，感觉他们的交互稿是很棒的，十分值得学习。

游戏策划，这里总结一下，包括：

  1. 游戏背景设计。游戏背景给玩家提供合合理的世界描述，使玩家在游戏过程中的行为受到限制并具有合理性。游戏世界观，包括：世界架构、人文地理、宗教信仰、政治、经济文化等的设定，用于突出情节、约束时空、明确风格。在世界观构架下，设定故事背景：关于游戏、游戏主角的故事背景。
  2. 游戏地图与场景设计。设计并不是设计出来，而是说明有什么地图场景。
  3. 游戏元素设计。主角，NPC，怪物（AI），道具。
  4. 任务与关卡设计。
  5. 游戏规则设计。包括数值策划，角色的属性等相关数值之间的平衡计算。另外还有战斗系统，换装系统等。
  6. 界面与用户控制。用户如何控制，界面显示什么？

最后就是将上面的总结成“游戏设计文档”，供团队使用。

四、程序

总体来说，程序到目前为止还是比较简单的。Flash air移动游戏开发，推荐学习开源游戏http://www.hungryherogame.com/，以及作者的视频教程http://www.hsharma.com/tutorials/。
几个值得注意的地方：

  1. 素材打包。
    由设计师及美工出了图（psd）后，交由我们切图，切图不想装ps的话推荐ImageReady，这是从ps独立出来的专门切片软件。
    然后使用TexturePacker打包到一张纹理集合以及它的xml，供starling框架使用texture,这样减少资源的加载。
  2. 游戏页面跳转。
    一直以为跳转是要remove元素的，实际上，和这想法不同，页面其实是刚开始就添加到舞台上了的，只是在某特定页面时，将其它页面所有元素visible属性为false，即将其隐藏起来，但其实是存在的。
    页面跳转是建一个NavigationEvent类（继承Event），自定义一个变量参数params，用于指定是哪个页面。通过事件侦听，switch判断params的值，作页面“跳转”。
    当点击按钮，如开始按钮，便派发事件（dispatchEvent），直接通知所有注册了的侦听器：params数值变成“play”了，然后作页面“跳转”操作。
    派发事件也就是事件调度，关于事件调度参照另一篇《Actionscript3事件调度与事件流》
  3. 暂停方法。
    暂停方法没什么参考，自己开始想到的就是通过中间变量储存暂停时的一些相关数值，然后继续游戏时再赋回来。这个笨方法显然到了后面越多数值时很难控制，所以只好想个另外的方法。后来发现其实并不用储存一些数值，因为数值的判断计算放在一个Enterframe事件上，只要把Enterframe事件remove掉，数值就会“暂停”在那个状态。而其他一些动画则可以控制movieclip的暂停即可。
    暂停时要注意的是，点击（触摸）继续按钮时，也算是一次触摸操作，如果把触摸侦听注册在舞台上的话，角色就会飞跃一次，所以需要一个“游戏区域”（矩形）用于确认点击（触摸）操作。
  4. 重力感应控制。
    重力感应用到flash.sensors.Accelerometer类，Accelerometer
    类根据由设备的运动传感器检测的活动调度事件。此数据表示设备的位置或沿三维轴的移动。当设备移动时，传感器会检测到此移动并返回加速数据。分为XYZ轴的加速，数值以
    G 为度量单位。（1 G 大约是 9.8 m/sec/sec。）。
    如何让重力感应更适合左右控制人物的操作，这很值得好好思考。
  5. 触摸控制。
    使用starling的Touch类，touch包括了移动设备的触摸及鼠标的操作，不好的一点就是刚开始时未点击鼠标，鼠标移入就当作touch。后来发现TouchPhase类，即：一个触碰，在它的生命周期内会遵循下面的过程：BEGAN(开始)
    -> MOVED(移动) ->
    ENDED(结束)，还有一个称之为HOVER(悬停)的状态，这种情况只会发生在使用鼠标的情况下。
    所以Touch侦听事件加入一个状态判断，就解决了鼠标点击问题和触摸灵敏太高的问题：

    private function onTouch(e:TouchEvent):void
    {
    var touch:Touch = e.getTouch(stage);
    phase = touch.phase;
    if (phase == “ended”)
    {
    //触摸后的处理

    }
    }
  6. 道具设置。
    道具设置暂时采用一个数组（Vector）添加道具和遍历道具。
  7. 道具触碰。
    道具触碰开始是想用starling里displayObject的hitTest方法的，不过试了一整天也不行，最后干脆用回最原始的方法，判断两对象的距离作碰撞检测，妥妥的TAT。
  8. 自适应屏幕。
    适应分辨率参考StarlingManual:多屏幕分辨率的开发，通过拉伸来适应屏幕大小：
    //自适应屏幕分辨率

var screenHeight:int = stage.fullScreenHeight;

Starling学习笔记（二）

在官网可以知道，Starling是Sparrow 框架在ActionScript3领域的一个姐妹框架。基于stage3D的API开发，而Stage3D有能力去控制 OpenGL、DirectX这些电脑显卡驱动或OpenGL ES2这些手机显卡驱动，最终由这些显卡驱动程序去驱动GPU工作。也就是传说中的GPU显卡加速。

就有人会疑惑了，stage3D不是渲染3D内容的吗？为何用来做2D。这里用我们该如何使用Stage3D中的drawTriangles这个API来绘制类似MovieClip来解释。GPU的优势在于它绘制三角形极其高效，所以使用drawTriangles API将绘制出两个三角形，之后我们再从一个纹理（texture，你可以理解成“素材”的意思）上面采样（或取色）并使用一种称为UV映射*的机制来着色于这两个三角形上即可，最后我们将能够得到一个纹理化（textured，即着了色的）的方块，我们就用它来作为我们Sprite对象的外观。若是在每一帧都改变其纹理，我们就可以得到一个MovieClip对象了。

 

*UV 映射是一种纹理化对象的方法。它依赖于两个值：U 水平 (x) 值和 V 垂直 (y) 值。这两个值不是基于像素值，而是基于百分比。0 U 和 0 V 表示图像的坐上角，1 U 和 1 V 表示右下角
 

那么，现在开始创建舞台元素玩玩了，如上一篇所讲，首先是创建启动类，然后在Game类中创建一个显示对象，建什么好呢？不妨查查API，starling.display，有Button、Image、MovieClip、Quad（四边形）以及其他一些容器。这里，我们创建一个button玩玩。

Button:

一个简单的按钮由一张图片和一个可选的文本组成。

你可以分别为按钮构造函数传递up和down两种状态的纹理。 如果你没有指定一个down状态的纹理，按钮会采用默认的办法：被触碰的时候缩小一点。 此外，你可以在按钮上覆盖一个文本，要自定义这个文本，只需要提供和原来的基本相同的文本框即可。 你可以使用textBounds属性，将文本移动到一个指定的位置。

判断按钮是否被触碰，请使用triggered事件类型， 用这个事件来代替普通的触碰事件。也就是说，用户只要在松开手指或者鼠标之前，移动手指/鼠标离开按钮区域，就可以取消对该按钮的点击操作。
 

package
{
    import starling.display.Button;
    import starling.display.Sprite;
    import starling.textures.Texture;

/**
 * ...
 * @author SHENGbanX
 */
public class Game7 extends Sprite
{
    [Embed(source = "../assets/chicken.png")]
    private static const Chicken:Class;
    public function Game7()
    {
                    //通过Texture类中的静态方法fromBitmap来创建Texture纹理对象
        var buttonTexture:Texture = Texture.fromBitmap(new Chicken());
        var button:Button = new Button(buttonTexture, "Hello Starling");
        button.x = 200;
        button.y = 200;
                    addChild(button);
    }

}

}
自己用ps简单地画了一个小鸡作为按钮的图片纹理，加上文字。但这样文字刚好在图片上，所以要用textBounds属性，把文字移到一个地方。

查api可知textBounds:Rectangle，Rectangle是flash.geom包（包中包含几何图形类（如点、矩形和转换矩阵）以支持 BitmapData 类和位图缓存功能。）里的一个类。

而Rectangle 对象是按其位置（由它左上角的点 (x, y) 确定）以及宽度和高度定义的区域。

加上button.textBounds = (0,50,100,100);

但报错了，类型不对，Rectangle不是就4个数作属性吗？思来想去，还是去看看starling的源代码：

        /* The bounds of the textfield on the button. Allows moving the text to a custom position. /
        public function get textBounds():Rectangle { return mTextBounds.clone(); }
        public function set textBounds(value:Rectangle):void
        {
            mTextBounds = value.clone();
            createTextField();
        }

仔细研究了一下，发现还是应该是那样没错的，试了很多种方法，通过trace(button.textBounds)结果是(x=0, y=0, w=73, h=53),以这个做值或(x=0, y=0, width=73, height=53)也不行。

最后这样才可以：

button.textBounds = new Rectangle(0, 50, 73,53);

 

没有教程的时候，查API、源码等方法自己研究还是很不错的！学习Starling更需如此。

Starling学习笔记（一）

开学零零碎碎的时间，两个星期了，主要在学习Starling框架，给我的感觉很不错，易用，效率高。

刚开始还无从下手，都是在Starling中文站照着学习和查阅API文档（中文）。还有就是中文说明书，学着学着就大概知道怎么用了。

Starling框架让使用Stage3D技术(在flash player 11中引入)创建二维程序和游戏成为可能。 它实现了和Flash 一样的显示列表系统，同时使用先进的GPUs加速渲染。以下讲讲我的体会：

开发工具：Flash builder4.6(FB4.7用不习惯）, Flash Develop。（给我的感觉是fd写代码更快更方便一点）

Starling框架：到官网下载starling包，里面有starling库源码和swc库文件以及一些demo项目文件。

Starling第一步

将您的项目链接到Starling swc库：

FB：建立新项目，在项目里建一个libs文件夹，把swc文件拉到里面，然后项目-属性-构建路径里添加libs（swc文件夹/文件）。

FD：建立新项目，把swc文件拉到libs文件夹里，然后右键选择-添加到库。

建立启动类：

src源路径里添加新类（class）文件——作为starling启动类，继承自flash的Sprite（而下面Game类以及之后用到starling的都继承自starling的sprite）。代码如下：

package
{
    import flash.display.Sprite;
    import starling.core.Starling;

[SWF(frameRate = 60, width = 600, height = 1024, backgroundColor = 0xffffff)]
public class Startup extends Sprite
{    

    public function Startup()
    {
        var _starling:Starling = new Starling(Game, stage);
        _starling.start();
    }

}

}
上面用核心starling类创建了starling对象_starling，然后用start方法启动它，这样，他就会按照设置好的信息[SWF…]渲染Game类了，Display显示列表和as原本的名字基本相似，所以容易混淆（常常自动填写代码错误…），但starling使用了stage3D，运行速度快。starling类还有其他用处，例如我常常就会习惯的加上_starling.showStats = true;来显示一些FPS和内存占用信息。

查API文档可以知道，starling类还可以：

访问Starling对象

在你的程序里，你可以在任意时间通过静态方法Starling.current访问当前的Starling对象。它会返回 当前活跃的Starling实例（实际上大多数程序只会有一个Starling对象）。

观察口，视口

即是Starling内容渲染显示的范围，默认情况下是舞台大小。但是，你可以使用“viewport”属性去改变它。当你只想在屏幕的一部分中 渲染，或者播放器大小改变时，这个属性非常有用。随后你可以监听由Starling舞台发出的RESIZE事件。

原生Flash叠加层

有时你会想在Starling上面显示一些原生Flash内容，这就是nativeOverlay属性的作用。它返回的是在Starling之上 的原生FLash Sprite。你可以在叠加层内添加原生Flash对象。

但是需要注意，在3D加速内容之上的原生Flash内容在某些（移动）平台上会有性能上的问题。因为这个原因，请注意删除这一层中所有 不需要和不再使用的对象。当这一层被置空后Starling会从显示列表中删除原生Flash叠加层。

多点触摸

对于提供支持多点触摸功能的设备，Starling也支持这项功能。在开发过程中，我们大多数基本上都是使用鼠标的键盘，Starling可以 在“Shift”和“Ctrl”的帮助下（Mac苹果机是“Cmd”）模拟实现多点触摸事件。启动simulateMultitouch属性就可以激活 这个功能。

处理丢失的渲染内容

在某些系统和某些条件下（例如，从系统睡眠中返回），Starling的stage3D渲染内容会丢失。如果类属性“handleLostContext” 设置为 “true” Starling会自动修复丢失内容。但是需要注意，这是以提高内存消耗为代价的。Starling会在内存中创建纹理缓存，这样 在内容丢失时才可以修复它。

如果你想和丢失的内容交互，Starling在这个内容修复时发送”Event.CONTEXT3D_CREATE”事件，你可以在相对应的事件*中重新 创建这些不可用资源.

共享三维加速内容

正常情况下，Starling是独立处理stage3d加速内容。如果你想让Starling和其它的stage3d引擎同时使用，可能会达不到你需要的效果。 在这种情况下，你可以使用shareContext属性:

  1. 手动创建和定义一个让两个框架公用的context3D对象 (通过stage3D.requestContext3D 和 context.configureBackBuffer)创建.
  2. 使用已经定义了的context3d对象的stage3d实例来初始化Starling。 这样做会自动启用 shareContext.
  3. 调用Starling实例的 start()(与往常一样). 这样做是让 Starling 对事件监听进行排队处理 (keyboard/mouse/touch).
  4. 创建一个游戏逻辑（例如，使用原生的ENTER_FRAME事件）然后让它调用Starling的 nextFrame同时调用其它Stage3D引擎的同功能方法 . 相关类型的方法还有context.clear() 和 context.present().
     

接着新建一个Game类，

import starling.display.Sprite;
import starling.text.TextField;

public class Game extends Sprite
{
public function Game()
{
var textField:TextField = new TextField(400, 300, “Hello Starling!”);
addChild(textField);
}
}
一贯的hello world啦~然后把启动类Startup设为文档类，就可以测试了，一般都能测试成功了，当然，要注意调用的是flash的还是starling的方法。
