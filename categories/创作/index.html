Pulupulu手机游戏开发记录（二）——技术篇

第一次制作手机游戏，总的来说，给我的感觉还是不错的，从策划到美工，从设计到程序，其中遇到的困难也不少，这些困难包括技术上的，也包括团队上的。虽然同一个工作室，但自发团队免不了拖拉问题，一拖拉就拖慢了整个项目的进度，更严重的拖了太久就减淡整个团队的开发热情。另外还因为各自的空闲时间不一样，很难聚在一起，最多的只是在讨论组里讨论。
一个项目没了热情，结果就注定是个坑。才刚做到了beta0.1，后面还有东西要加。

技术上的话，困难也不大，但还是再做个技术相关的总结吧，横跨四个领域的总结！

一、策划

这次的策划是平面组的bobo和ck，虽然说是策划，但他们也不懂得该怎么做。

“策划也不懂该干什么，相对于策划，其实我更希望能做有关设计或者画画的东西，只是能力不够而已。”——bobo

作为项目总监，我根据我已知的制作游戏项目需要，分配给他们策划的任务：将游戏流程转化为“线框图”，整体完善游戏机制、玩法。其中还包括一些小讨论：输赢机制、游戏道具的添加、有新意的玩法等等。

将游戏流程转化为线框图，这很重要，即对游戏策划的整体框架的确立，也帮助了程序员架构游戏程序。制作时是参考江南大学交互工作室的一些交互稿的，感觉他们的交互稿是很棒的，十分值得学习。

游戏策划，这里总结一下，包括：

  1. 游戏背景设计。游戏背景给玩家提供合合理的世界描述，使玩家在游戏过程中的行为受到限制并具有合理性。游戏世界观，包括：世界架构、人文地理、宗教信仰、政治、经济文化等的设定，用于突出情节、约束时空、明确风格。在世界观构架下，设定故事背景：关于游戏、游戏主角的故事背景。
  2. 游戏地图与场景设计。设计并不是设计出来，而是说明有什么地图场景。
  3. 游戏元素设计。主角，NPC，怪物（AI），道具。
  4. 任务与关卡设计。
  5. 游戏规则设计。包括数值策划，角色的属性等相关数值之间的平衡计算。另外还有战斗系统，换装系统等。
  6. 界面与用户控制。用户如何控制，界面显示什么？

最后就是将上面的总结成“游戏设计文档”，供团队使用。

上面列了这么多，其实最重要的就是数值策划和游戏背景（故事）策划。

二、美工

先说一个奇怪的现象：设计师普遍不喜欢被叫作美工，而程序员从不在乎被叫作码农。

其中的道理，大概只有我这种喜欢设计又喜欢代码的才懂的吧…

1、所谓设计和美工是有较大区别的，设计师到了一定程度都会倾向于设计(design)这个高端、优雅的名词，而认为美工是涂涂画画，顶多做做游戏人设的。所以傲娇的设计师一旦被叫美工，就瞬觉自己身价掉了不止一点半点，当然不喜欢啦~
2、而程序员，虽然苦逼，但也享受着苦逼的过程与结果，而码农正对了这苦逼的感觉，所以，程序员也不会在乎这名词了。
3、最最最重要的一点是，设计师多是女生，而程序员多是男生… 回到正题。

美工yano的画风如其人，是清新型的，让她画猥琐风感觉很受罪…最后画出来的已经很不错了！主要是效率高，赞！

四、程序

总体来说，程序到目前为止还是比较简单的。Flash air移动游戏开发，推荐学习开源游戏http://www.hungryherogame.com/，以及作者的视频教程http://www.hsharma.com/tutorials/。
几个值得注意的地方：

  1. 素材打包。
    由设计师及美工出了图（psd）后，交由我们切图，切图不想装ps的话推荐ImageReady，这是从ps独立出来的专门切片软件。
    然后使用TexturePacker打包到一张纹理集合以及它的xml，供starling框架使用texture,这样减少资源的加载。
  2. 游戏页面跳转。
    一直以为跳转是要remove元素的，实际上，和这想法不同，页面其实是刚开始就添加到舞台上了的，只是在某特定页面时，将其它页面所有元素visible属性为false，即将其隐藏起来，但其实是存在的。
    页面跳转是建一个NavigationEvent类（继承Event），自定义一个变量参数params，用于指定是哪个页面。通过事件侦听，switch判断params的值，作页面“跳转”。
    当点击按钮，如开始按钮，便派发事件（dispatchEvent），直接通知所有注册了的侦听器：params数值变成“play”了，然后作页面“跳转”操作。
    派发事件也就是事件调度，关于事件调度参照另一篇《Actionscript3事件调度与事件流》
  3. 暂停方法。
    暂停方法没什么参考，自己开始想到的就是通过中间变量储存暂停时的一些相关数值，然后继续游戏时再赋回来。这个笨方法显然到了后面越多数值时很难控制，所以只好想个另外的方法。后来发现其实并不用储存一些数值，因为数值的判断计算放在一个Enterframe事件上，只要把Enterframe事件remove掉，数值就会“暂停”在那个状态。而其他一些动画则可以控制movieclip的暂停即可。
    暂停时要注意的是，点击（触摸）继续按钮时，也算是一次触摸操作，如果把触摸侦听注册在舞台上的话，角色就会飞跃一次，所以需要一个“游戏区域”（矩形）用于确认点击（触摸）操作。
  4. 重力感应控制。
    重力感应用到flash.sensors.Accelerometer类，Accelerometer
    类根据由设备的运动传感器检测的活动调度事件。此数据表示设备的位置或沿三维轴的移动。当设备移动时，传感器会检测到此移动并返回加速数据。分为XYZ轴的加速，数值以
    G 为度量单位。（1 G 大约是 9.8 m/sec/sec。）。
    如何让重力感应更适合左右控制人物的操作，这很值得好好思考。
  5. 触摸控制。
    使用starling的Touch类，touch包括了移动设备的触摸及鼠标的操作，不好的一点就是刚开始时未点击鼠标，鼠标移入就当作touch。后来发现TouchPhase类，即：一个触碰，在它的生命周期内会遵循下面的过程：BEGAN(开始)
    -> MOVED(移动) ->
    ENDED(结束)，还有一个称之为HOVER(悬停)的状态，这种情况只会发生在使用鼠标的情况下。
    所以Touch侦听事件加入一个状态判断，就解决了鼠标点击问题和触摸灵敏太高的问题：

    private function onTouch(e:TouchEvent):void
    {
    var touch:Touch = e.getTouch(stage);
    phase = touch.phase;
    if (phase == “ended”)
    {
    //触摸后的处理

    }
    }
  6. 道具设置。
    道具设置暂时采用一个数组（Vector）添加道具和遍历道具。
  7. 道具触碰。
    道具触碰开始是想用starling里displayObject的hitTest方法的，不过试了一整天也不行，最后干脆用回最原始的方法，判断两对象的距离作碰撞检测，妥妥的TAT。
  8. 自适应屏幕。
    适应分辨率参考StarlingManual:多屏幕分辨率的开发，通过拉伸来适应屏幕大小：
    //自适应屏幕分辨率
var screenWidth:int = stage.fullScreenWidth;
var screenHeight:int = stage.fullScreenHeight;
var viewPort:Rectangle = new Rectangle(0, 0, screenWidth, screenHeight);
_st = new Starling(Game, stage, viewPort);

Pulupulu手机游戏开发记录（一）

由于这两年来flash开发界的变化较大，html5时代的来临，让工作室Flash组面临着没有什么项目可做的困境，工作室flash组还是否需要存在?这个问题也让我这个flash组组长感到困扰。

当然我是喜欢flash的，也坚信flash的优势。于是，在我这届，我决定改变flash组的方向，由原本的前端开发转变为现在的前端+移动端开发，并且应用范围也由原来的web动态效果转变为现在的web应用及游戏开发。这样以适应如今flash开发界的趋势。

于是，便有了这个pulupulu这个手机游戏项目。项目开始到现在已有一个月了，终于在今天做出了beta 0.1版本，但只是一个demo。这里做一下这一个月来的开发记录，也算是一个中期经验总结吧。

Pulupulu

游戏策划

  • 主题：在天朝吃东西，难免食后腹泻肚痛，特殊情况不能上厕所，想放屁？难受难忍，那就尽情地放吧！
  • 背景：课堂上突然肚痛想放屁，却不能，苦逼，于是思绪yy飞出向上飞，吃番薯储存“屁能”，用来释放屁。途中捡道具、捡厕纸，收集黑宝石（打开异次元空间解决生理问题），收集厕纸可提高分数、买东西、提升技能等。直到飞到终点：梦想的厕所！
  • 玩法：触摸放屁，放屁产生反冲力向上飞，重力左右控制。
  • 道具：有番薯、辣椒、小鸟、厕纸、黑宝石等。
  • 输赢机制：高低决定分数，掉到屏幕底线以下且黑宝石数为零则输；
  • 设计风格：预期是找你妹的猎奇猥琐风格，但还是美工说的“画出什么就什么啦”。

特点

  1. 风格和谐，口味略重，关于放屁的游戏。
  2. 玩法简单，轻松休闲。
  3. 利用道具技能升级，人物换装，具有挑战性。
  4. 不同的食物会有不同颜色的屁，传说能放出彩虹屁。
  5. 工科男放屁是一门艺术，当达到一定的境界就能放出诸如N阶矩阵屁、向量屁、地图屁（参照地图炮）、三次元屁（相对于二次元）等。
  6. 结合中国传统文化，诸如周易、道家、儒学等。
  7. ……

开发细则

  1. 使用Actionscript3开发，结合Starling框架以及DragonBones骨骼方案
  2. Adobe Air跨平台开发（android,ios,rim)
  3. 使用Flash Develop开发工具
    3月27日，我正式提出了要做手机游戏这个想法，在此前我用了4周时间学习starling框架，并研究了一些游戏、手游的架构、策划、制作等知识，就等技术自认为差不多成熟的时候我提出了这想法。我当时就打算我只写程序，把美工、设计让出来，虽然我感觉自己能胜任，但一方面要形成团队合作的氛围，另一方面在手游设计上要锻炼一下工作室大一的同学。

当天晚上，我召集了对我这个手游感兴趣的平面组大一的3个组员，在工作室讨论了游戏的整体策划及简单架构，人员的确立与相关的工作计划。

策划：ck、bobo、程序：我、设计：ck、美工：LinkYano，当然，项目总监就是我啦~。

这是工作室第一次做手游，毫无经验，计划做个玩法简单点的手机游戏，并进行“微创新”。即：简单而不简单。并实现跨平台（pc+android+ios）。
第一周，我主要还是边学习边研究，并统筹整个项目的任务分配。并让flash组大一的ms加入，我带带他，且先让他熟悉一下starling框架。

3月28日，想好代码实现的整体思路；为整个项目设定设计规范，以适应各分辨率尺寸屏幕，这里我决定以720*1280屏幕大小进行设计。

3月29日，考虑如何适应各种分辨率屏幕，适应分辨率参考StarlingManual:多屏幕分辨率的开发，通过拉伸来适应屏幕大小。

第二周，没有UI单单测试代码是十分无趣的，在我一催再催之下，ck终于出了UI稿了，泪流满面的同时，开始正式写程序了。

4月9日，游戏首页的UI初稿出来了，欣喜之下提出了一些意见。并于次日修改了。

4月11日，我开始切图写游戏界面。

4月13日，完成了游戏首页、游戏进行界面、暂停界面，并实现了之间的切换。

4月14日，开始写游戏进行程序。完成了触摸控制飞跃和重力感应控制人物左右移动。LinkYano把icon画好了。

第三周，这周我由于周五考试要复习，就放下了这个游戏5天，转交给ms继续并指导一下他。

第四周，状态全开写游戏进行程序。

4月22日，写完了“放屁”功能。

要感谢trello，进行这个项目的管理，并让我有机会作这样的流水账记录:)

五一放假了，暂时告一段落。虽然中途有点坑，但我还是依然坚持。即使每天做这个东西的时间不多，中间还缺了一个星期，还有上星期4月21日老师要做个视频，被Ae搞烦了几天，最后跟老师说实在完不成，然后开心地全身心转战这游戏。这个星期，相当努力。几乎全身心都放在这个游戏上了。尤其写拾取道具的碰撞检测方法时，从下午2点到晚上11点，由于尝试starling的碰撞方法出现种种问题，几近崩溃。但在最后终于写好的时候几乎热泪盈眶了。

这个项目是我负责的，对于大一各位的表现值得表扬，虽然他们当时答应下来只是随随便便地。LinkYano妹子最赞了，美工稿出得快，连icon也一教就会。还有ms虽然刚接触AS3没多长时间，但进步却是很快的，相信接下来能够帮上较大的忙~

编程效率是个问题，我觉得自己是渣渣的，但有时状态一来就很快很高效。我认为编程速度更多的取决于你的思维逻辑，以及知识的积累。这点我体会很深。

这项目到此，并不是结束，后续迭代以及进一步详细策划、修改优化UI设计的工作还等着我们，我们还要继续努力！
